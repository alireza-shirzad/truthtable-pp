\subsection{Fingerprinting}

A string fingerprint can be understood as the answer vector to a small
collection of yes/no questions asked of a string. In the most precise
form one could imagine, there would be one such question per
\emph{atomic feature} of interest---for instance, ``does this string
contain the letter \texttt{a}?'', ``does it contain the bigram
\texttt{th}?'', ``does it start with a digit?'', or any other Boolean
property one cares to evaluate. Evaluating every atomic feature gives a
lossless, but potentially very large, summary of the string.

The fingerprint trades this precision for size. Instead of storing one
bit per atomic feature, we partition the features into $n$ groups and
\emph{or}~them together: a single bit fires whenever \emph{any} of its
constituent features fires. Each bit of the fingerprint is therefore
not a single feature but the disjunction of many features. This
compression is what makes fingerprints small and fast to evaluate---and
also what makes them lossy. The design problem is then twofold: choose
the family of atomic features to draw from, and choose how to group
them into bits.

Let $\mathcal{F} = \{f_1, f_2, \dots, f_m\}$ be a finite family of
\emph{atomic features}, where each $f_i$ is a Boolean function on
strings:
\begin{equation*}
  f_i : \Sigma^{*} \to \{0,1\}.
\end{equation*}
We make no assumption about what each $f_i$ computes; it may test for
the presence of a character, the presence of an $n$-gram, a positional
condition, the output of a hash function, or any other property.

A fingerprint of bitwidth $n$ is induced by a partition
$\{B_1, \dots, B_n\}$ of $\mathcal{F}$. The $j$-th bit of the
fingerprint of a string $s$ is the disjunction of the features assigned
to bin $B_j$:
\begin{equation}
  \Vector{f}_s[j] \;=\; \bigvee_{f \in B_j} f(s).
  \label{eq:bit-as-or}
\end{equation}
The full fingerprint $\Vector{f}_s \in \{0,1\}^n$ is the vector of these
disjunctions.

\paragraph{Example.} Let $\Sigma = \{a, e, h, l, n, t, u\}$ and take
the family $\mathcal{F} = \{f_a, f_e, f_h, f_l, f_n, f_t, f_u,
g_{\text{th}}, g_{\text{ut}}\}$, where $f_c(s) = \mathbb{1}[c \in s]$
and $g_{xy}(s) = \mathbb{1}[xy \text{ occurs in } s]$. With bitwidth
$n = 3$ and partition $B_1 = \{f_a, f_l, f_u\}$, $B_2 = \{f_e, f_h,
f_n, f_t\}$, $B_3 = \{g_{\text{th}}, g_{\text{ut}}\}$, the
fingerprints of a few strings are:
\begin{center}
  \begin{tabular}{lccc}
    \toprule
    String & $\Vector{f}[1]$ & $\Vector{f}[2]$ & $\Vector{f}[3]$ \\
    \midrule
    \texttt{utn}   & 1 & 1 & 1 \\
    \texttt{lute}  & 1 & 1 & 1 \\
    \texttt{heat}  & 1 & 1 & 0 \\
    \texttt{ten}   & 0 & 1 & 0 \\
    \bottomrule
  \end{tabular}
\end{center}
For instance, \texttt{lute} sets $\Vector{f}[3] = 1$ via the bigram \texttt{ut};
the fingerprint records that \emph{some} feature in $B_3$ fired, but
not which.


\subsection{Approximate Filtering}

Fingerprints become useful when we want to quickly decide whether a
candidate string $s$ could possibly contain a pattern $\zeta$, without
performing the full containment test.

\paragraph{The pruning rule.} For a pattern $\zeta$ and a candidate string
$s$, the fingerprint check accepts $s$ iff
\begin{equation}
  \Vector{f}_\zeta \subseteq \Vector{f}_s
  \;\;\Longleftrightarrow\;\;
  \forall j \in [n]:\; \Vector{f}_\zeta[j] = 1 \Rightarrow \Vector{f}_s[j] = 1.
  \label{eq:subset-check}
\end{equation}
For this check to be a sound filter---never rejecting a true
match---the feature family $\mathcal{F}$ must satisfy a
\emph{monotonicity} condition: whenever $\zeta$ is contained in $s$ in the
sense the query intends, every feature satisfied by $\zeta$ must also be
satisfied by $s$,
\begin{equation}
  \zeta \sqsubseteq s \;\;\Longrightarrow\;\; \forall f \in \mathcal{F}:\;
  f(\zeta) \le f(s).
  \label{eq:monotonicity}
\end{equation}
Under~\eqref{eq:monotonicity}, the subset
check~\eqref{eq:subset-check} cannot produce false negatives.

\paragraph{Where false positives come from.} The converse of the
subset check fails because of the \emph{or}-compression
in~\eqref{eq:bit-as-or}. A string $s$ may light bit $j$ via a feature
$f \in B_j$ unrelated to those the pattern requires; the fingerprint
cannot tell which feature of $B_j$ caused the bit to fire. Two design
choices govern how often this happens:
\begin{enumerate}
  \item The \textbf{feature family} $\mathcal{F}$. Richer families
  (e.g., $n$-grams, positional features) carry more information per
  atomic feature and therefore tolerate coarser bin assignments, at
  the cost of a larger $|\mathcal{F}|$ and a more expensive
  partition-selection problem.
  \item The \textbf{partition} $\{B_1, \dots, B_n\}$. Grouping
  features the workload needs to \emph{distinguish} into the same bin
  destroys that information; grouping interchangeable features wastes
  little.
\end{enumerate}

\paragraph{Example.} Reusing the setup from above, take pattern $\zeta =
\texttt{utn}$ with $\Vector{f}_\zeta = (1, 1, 1)$. Then \texttt{lute} passes the
filter as a false positive (its $\Vector{f}[3]$ bit is lit by \texttt{ut} rather
than by \texttt{utn}), while \texttt{heat} is correctly rejected on
$\Vector{f}[3]$ and \texttt{ten} on $\Vector{f}[1]$. Replacing $B_3$ with another
character bin would weaken the filter on \texttt{heat}; merging
\texttt{u} and \texttt{t} into one bin would weaken it on
\texttt{ten}.

\subsection{Exact Filtering}

Let $\col$ be a string column in the database containing strings of maximum length $l$. We encode $\col$ in as a tuple of 4 columns, $( \mathsf{h},\mathsf{b},\mathsf{a_{hb}}, \mathsf{d}, \mathsf{a_d}, \mathsf{s_b})$, with the following semantics:
\begin{itemize}[nosep]
    \item Hash column $\mathsf{h}$ is of the same length as $\col$ containing the hash of each string in $\col$.
    \item Data column $\mathsf{d}$ is of the size $l \cdot |\col|$, containing the concatenation of all strings in $\col$.
    \item Boundary column $\mathsf{b}$ is a column of the same length as $\col$ with $\mathsf{b}[i]$ being the index of the first character of the $i$-th string in $\col$ in the data column $\mathsf{d}$.
    \item The data activator column $\mathsf{a_d}$ is of the same length as $\mathsf{d}$, with $\mathsf{a_d}[i]$ being $1$ if the $i$-th character in $\mathsf{d}$ is an active chatacter of the column $\col$.
    \item The hash/boundry activator column $\mathsf{a_{hb}}$ is of the same length as $\col$, with $\mathsf{a_{hb}}[i]$ being $1$ if the $i$-th string in $\col$ is active.
    \item The boundary selector column $\mathsf{s_b}$ is of the same length as $\mathsf{b}$, with $\mathsf{s_b}[i]$ being $1$ if the $i$-th boundary in $\mathsf{b}$ is active.
    \end{itemize}

 
    \subsubsection{Prefix Check} In a prefix check protocol, the verifier is given a tuple of the form $( \mathsf{h},\mathsf{b},\mathsf{a_{hb}^{old}}, \mathsf{d}, \mathsf{a_d^{old}}, \mathsf{s_b})$  and new activators $\mathsf{a_{hb}^{new}}$ and $\mathsf{a_d^{new}}$. The prover needs to convince the verfier that the tuple $( \mathsf{h},\mathsf{b},\mathsf{a_{hb}^{new}}, \mathsf{d}, \mathsf{a_d^{new}}, \mathsf{s_b})$ is the result of applying a prefix filter $\mathsf{str}\%$ on the original tuple where $\mathsf{str}$ is the prefix string of length $k$.
    We present the protocol in the following steps:

    \parhead{False Positive Check} To make sure that the active elements of $\mathsf{a_d^{new}}$ correctly match the pattern, the prover and the verifier run $k$ instances of zerocheck PIOP on the columns $[(\mathsf{d}-\mathsf{str}[0], \mathsf{s_b^{i}}\cdot \mathsf{a_d^{new}})]_{i=0}^{k-1}$ where $\mathsf{s_b^{i}}$ is the $i$-th cyclic rotatiion of $\mathsf{s_b}$, and $\mathsf{str}[i]$ is the $i$-th character of $\mathsf{str}$. Independently, the prover and the verifier also run $l$ instances of rotation-check piops on the columns $[ \mathsf{s_b^{i}} ]_{i=0}^{k-1}$ to make sure that the boundary selector column is correctly rotated.

    \parhead{False Negative Check} To make sure that all the inactive elements of $\mathsf{a_d^{new}}$ are correctly filtered out, the prover and the verifier run $k$ instances of NoZeroCheck PIOP on the columns $[(\mathsf{d}-\mathsf{str}[0], \mathsf{s_b^{i}}\cdot (1-\mathsf{a_d^{new}}))]_{i=0}^{k-1}$.

    \parhead{Activator well-formedness Check} To make sure that the new activators $\mathsf{a_{hb}^{new}}$ and $\mathsf{a_d^{new}}$ are well-formed, the prover and the verifier run two instances of binary check on the polynomials $\mathsf{a_{hb}^{new}}$ and $\mathsf{a_d^{new}}$.

    \parhead{Activator consistency Check} To make sure that the new activators $\mathsf{a_{hb}^{new}}$ and $\mathsf{a_d^{new}}$ are consistent with each other, the prover convinces the verifier that the following multiset equality holds:
    \begin{align*}
      \bigcup_{i=0}^{l-1} (\mathsf{b}+i, \mathsf{a_{hb}^{new}}) = (\idx, \mathsf{a_d^{new}})
      \end{align*}

    \begin{align*}
      (\mathsf{b}, \mathsf{a_{hb}^{new}}) = (\idx, \mathsf{s})
